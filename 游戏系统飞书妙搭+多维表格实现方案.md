# 游戏系统飞书妙搭 + 多维表格后端数据收集实现方案

## 📋 项目概述

本项目实现一个基于飞书妙搭和多维表格的游戏数据收集系统，通过 `postMessage` 实现前端游戏页面与妙搭应用之间的双向通信，完成学生身份信息收集、游戏进度数据的持久化存储。

### 三步核心流程

#### 🔹 第一步：学生填写身份信息（妙搭 → 游戏）

| 序号 | 操作 | 发生位置 | 做了什么 |
|------|------|---------|---------|
| 1 | 学生在妙搭左侧面板填写：学号、姓名、班级 | 妙搭应用 | 学生输入信息 |
| 2 | 点击「保存信息」按钮 | 妙搭应用 | ① 存到妙搭的localStorage<br>② 通过`postMessage`发送给iframe里的游戏 |
| 3 | 游戏页面接收消息 | 游戏页面 | ① 收到`FEISHU_STUDENT_INFO`消息<br>② 存到游戏页面的localStorage<br>③ 按钮文字从「请填写信息」→「提交成绩到飞书」 |

**核心代码（妙搭发消息）**：
```javascript
iframe.contentWindow.postMessage({
    type: 'FEISHU_STUDENT_INFO',
    studentId: '学号',
    studentName: '姓名',
    className: '班级'
}, '*');
```

#### 🔹 第二步：学生玩游戏 + 提交成绩（游戏 → 妙搭）

| 序号 | 操作 | 发生位置 | 做了什么 |
|------|------|---------|---------|
| 1 | 学生玩游戏，完成关卡 | 游戏页面 | 游戏内产生进度数据 |
| 2 | 点击右下角「📤 提交成绩到飞书」按钮 | 游戏页面 | 收集当前游戏数据 |
| 3 | 按钮显示「🔄 正在同步...」 | 游戏页面 | 发送postMessage给妙搭 |

**收集的数据格式**：
```javascript
{
    studentId: '学号',        // 从localStorage取
    studentName: '姓名',      // 从localStorage取
    className: '班级',        // 从localStorage取

    submitTime: '2026-05-07...'
}
```

**核心代码（游戏发消息给妙搭）**：
```javascript
window.parent.postMessage({
    type: 'FEISHU_GAME_SUBMIT',
    data: 上面收集的数据对象
}, '*');
```

#### 🔹 第三步：妙搭接收数据并写入飞书表格（妙搭 → 多维表格）

| 序号 | 操作 | 发生位置 | 做了什么 |
|------|------|---------|---------|
| 1 | 妙搭监听`message`事件 | 妙搭应用 | 收到`FEISHU_GAME_SUBMIT`消息 |
| 2 | 解析数据 | 妙搭应用 | 拿到游戏提交的所有数据 |
| 3 | 调用飞书多维表格API | 妙搭应用 | 同时写入3个表 |

**写入的三个数据表逻辑**：

| 表名 | 写入逻辑 | 关键字段 |
|------|---------|---------|
| **学生信息表** | 有记录则更新，无则新增 | 学号、姓名、班级、最后登录时间 |
| **游戏进度总览** | 按「学号+游戏编号」查找，有则更新，无则新增 | 学号、游戏编号、当前关卡、得分、最后游玩时间 |
| **关卡完成记录** | 每次提交都新增一条记录 | 所有字段都存下来，便于统计分析 |

## 💡 第三步：实现方案设计

### 方案对比分析

#### 方案A：前端直接调用飞书开放API
**实现方式**：每个游戏的前端页面，完成任务后直接调用飞书多维表格API写入数据。

| 优点 | 缺点 |
|------|------|
| 架构简单，无需中间层 | API key暴露在前端代码，严重安全隐患 |
| 开发快速，直接调用 | 19个游戏都需要重复写API调用逻辑 |
| 数据链路短 | 跨域问题（游戏在github.io，API在feishu.cn） |
| | 每个游戏都要维护API凭证 |

**风险评估**：❌ 不推荐。安全风险过高，API密钥可能被反编译获取。

---

#### 方案B：飞书妙搭 + webhook（推荐方案）

**实现方式**：使用飞书妙搭作为中间层，前端发送数据到妙搭的webhook，再同步到多维表格。

| 优点 | 缺点 |
|------|------|
| ✅ 安全：API token存储在妙搭后端，不暴露前端 | 需要配置妙搭流程（一次配置，永久使用） |
| ✅ 有可视化界面，便于调试监控 | 初期配置有一定学习成本 |
| ✅ 统一入口，19个游戏共用一套逻辑 | |
| ✅ postMessage跨域通信解决跨域问题 | |
| ✅ 妙搭可直接访问飞书生态，无需额外认证 | |
| ✅ 支持失败重试和日志记录 | |

**推荐理由**：本项目的设计文档已采用此方案，且完美解决了安全、跨域、统一管理等核心问题。

---

#### 方案C：统一数据收集脚本

**实现方式**：创建一个统一的JS函数，所有游戏调用这个函数来发送数据。

| 优点 | 缺点 |
|------|------|
| 代码复用，易于维护 | 还是需要处理API安全问题 |
| 统一数据格式规范 | 需要额外的后端服务作为代理 |
| | 增加了部署和运维复杂度 |

    showStatus('✅ 信息已保存并发送', 'success');
    console.log('📨 学生信息已发送到游戏:', studentInfo);
}

// 处理来自游戏的消息
async function handleGameMessage(event) {
    const message = event.data;
    
    if (message.type === 'FEISHU_GAME_SUBMIT') {
        console.log('📥 收到游戏提交的数据:', message.data);
        showStatus('🔄 正在写入飞书表格...', 'info');
        
        try {
            // 并行写入3个表格
            await Promise.all([
                upsertStudentInfo(message.data),
                upsertGameProgress(message.data),
                createLevelRecord(message.data)
            ]);
            
            showStatus('✅ 所有数据已成功写入飞书表格', 'success');
            console.log('✅ 所有表格写入完成');
            
        } catch (error) {
            console.error('❌ 写入表格失败:', error);
            showStatus('❌ 写入失败: ' + error.message, 'error');
        }
    }
}

// ========== API辅助函数 ==========

// 创建记录
async function createRecord(tableId, fields) {
    // 妙搭调用飞书多维表格API
    // 实际API调用根据妙搭文档实现
    return await lark.bitable.appTableRecord.create({
        path: {
            app_token: BASE_CONFIG.baseToken,
            table_id: tableId
        },
        data: { fields }
    });
}

// 更新记录
async function updateRecord(tableId, recordId, fields) {
    return await lark.bitable.appTableRecord.update({
        path: {
            app_token: BASE_CONFIG.baseToken,
            table_id: tableId,
            record_id: recordId
        },
        data: { fields }
    });
}

// 根据单个字段查找记录
async function findRecordByField(tableId, fieldName, fieldValue) {
    const result = await lark.bitable.appTableRecord.list({
        path: {
            app_token: BASE_CONFIG.baseToken,
            table_id: tableId
        },
        params: {
            filter: `CurrentValue.[${fieldName}] = "${fieldValue}"`
        }
    });
    
    const records = result.data?.items || [];
    return records.length > 0 ? records[0] : null;
}

// 根据两个字段查找记录
async function findRecordByTwoFields(tableId, field1, value1, field2, value2) {
    const result = await lark.bitable.appTableRecord.list({
        path: {
            app_token: BASE_CONFIG.baseToken,
            table_id: tableId
        },
        params: {
            filter: `CurrentValue.[${field1}] = "${value1}" && CurrentValue.[${field2}] = "${value2}"`
        }
    });
    
    const records = result.data?.items || [];
    return records.length > 0 ? records[0] : null;
}

### 4. 常见问题排查
